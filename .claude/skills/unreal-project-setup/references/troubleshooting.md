# Troubleshooting project setup (UE 5.6/5.7)

Doc-sourced from `references/api/` — not compile-tested here (no engine installed).

## `unreal.sh` exits 127 — "no Unreal Engine install found"

Expected **in this environment**: no engine is installed, so the shared wrapper
fails fast with install guidance instead of pretending to build. The scaffold
(`new_unreal_project.sh`) is **file-only** and works without an engine; only the
`gencproj` / `build` / `editor` / `uat` subcommands need one. To enable them:

```bash
export UE_ROOT=/path/to/UnrealEngine    # the dir that contains Engine/
scripts/unreal.sh root                  # should print the resolved root
```

## UBT doesn't see my module / "no rules object" / module missing from build

The single most common cause: **a C++ module with no `.Build.cs`**. UBT discovers
modules by their `.build.cs` (a `ModuleRules` subclass next to the source), not by
folder presence. Checklist:

- The module folder has a `<Module>.Build.cs` whose **class name == file base name
  == folder name** (e.g. `Source/MyGame/MyGame.Build.cs` → `public class MyGame :
  ModuleRules`).
- The module is listed in the `.uproject` `"Modules"` array with a `"Type"`.
- The relevant `.Target.cs` adds it: `ExtraModuleNames.Add("MyGame");` (Editor-only
  modules go only in `*Editor.Target.cs`).
- You re-ran **GenerateProjectFiles** after adding it (and rebuilt). Per the docs,
  "If you make local changes to modules such as adding or removing source files,
  you should re-run GenerateProjectFiles."

## Crash at runtime with cooked data, hard to debug

Often a bad module name in the implement macro. Per *Editor Modules*: "The name
you provide in the second field for `IMPLEMENT_GAME_MODULE` must be valid,
otherwise your project will crash at runtime with cooked data… Make sure the name
matches the module's folder name." The same applies to
`IMPLEMENT_PRIMARY_GAME_MODULE` / `IMPLEMENT_MODULE` — the module-name argument
must match the folder/`.Build.cs` name exactly.

## I committed `Saved/` / `Intermediate/` / `Binaries/` / `DerivedDataCache/`

These are all **regenerated** (UE *Directory Structure*: "These files can be
deleted and rebuilt."). Never commit them; the bundled `assets/unreal.gitignore`
excludes them. If already committed, untrack them:

```bash
git rm -r --cached Saved Intermediate Binaries DerivedDataCache
git add .gitignore && git commit -m "stop tracking regenerated UE dirs"
```

## I gitignored `Content/` — now art/levels/Blueprints are missing on clone

`Content/` is the opposite: it holds **binary** `.uasset`/`.umap` files that ARE
your assets and Blueprints. **Commit `Content/`.** Do NOT add it (or `Config/`,
`Source/`, the `.uproject`) to `.gitignore`. For large binaries use Git LFS, but
they still must be tracked. (The fastest way to lose all your art on a fresh clone
is a stray `Content/` in `.gitignore`.)

## IDE shows errors but UBT builds fine (or vice-versa)

The `.sln`/IntelliSense data is an *intermediate* generated by GenerateProjectFiles
and can drift from reality — "the build process executes independently of any
project files." If IntelliSense is stale, re-run `gencproj`. If a build fails but
the IDE looks fine, trust the build output: UBT reads `.Build.cs`/`.Target.cs`, not
the `.sln`.

## No `.sln` / IDE project files at all

Run `scripts/unreal.sh gencproj -project="<dir>/<Name>.uproject" -game`. The
scaffold deliberately does **not** generate these (they need an engine, are
machine-specific, and are gitignored). They are created on demand.

## Black screen when I run a packaged build

`Config/DefaultEngine.ini` has no `GameDefaultMap`. Per *Packaging*: "Every
project must set a Game Default Map prior to packaging — without it, there's
nothing for the project to load at runtime." Set
`[/Script/EngineSettings.GameMapsSettings] GameDefaultMap=/Game/Maps/YourMap`.

## "Blueprint vs C++" — which is my project?

If there's a `Source/` dir with `.Build.cs` and the `.uproject` lists `"Modules"`,
it's a C++ project. No `Source/` and no `"Modules"` → Blueprint-only. Adding the
first C++ class to a Blueprint project is what makes UE generate the IDE solution
(see `Set_Up_and_Compile_Your_Project.md`). This suite assumes C++.

## DebugGame configuration unavailable

DebugGame "is not available for Blueprint-only projects" (*Packaging* table). You
need a C++ project (this skill's scaffold gives you one).

## Plugin not loading / not found

UBT finds plugins by their `.uplugin`; the plugin must be under
`<ProjectRoot>/Plugins/<Name>/` (or `Engine/Plugins/`). A plugin with code needs a
`Source/` folder containing a `.Build.cs`; one without a `Source/` is skipped by
the project generator but still loaded at startup if binaries exist. Enable it per
project in the `.uproject` `"Plugins"` array (or **Edit > Plugins** in-editor), and
respect the dependency hierarchy: a plugin can only depend on plugins/modules at
its level or higher.
