# Unity Project Setup (C# / Unity 6)

This skill gets you from nothing to a **Unity 6 (6000.x)** project that opens,
compiles C#, and is correctly committed to version control — driven from the
command line. Everything here is **doc-sourced from the embedded Unity 6 manual
(`references/api/`) and compile-verified against Unity 6000.4.10f1**: the
`new_unity_project.sh` scaffold was imported + compiled headless
(`unity.sh -batchmode -nographics -quit -projectPath …`, exit 0), and the
scripts generated by the sibling skills compiled clean on top of it. Where the
docs print a format or command verbatim, this skill copies it rather than
inventing it. (`scripts/unity.sh` resolves the Editor via `$UNITY_PATH` / Hub
locations / PATH and exits 127 with install instructions if none is found.)

## The one thing that trips everyone up

**`.meta` files are sacred; the `Library/` folder is disposable — and people get
this exactly backwards.** Unity creates a `.meta` for *every* file and folder
under `Assets/`. Each one holds that asset's unique GUID and import settings, and
Unity references assets by GUID internally. From the manual (`AssetMetadata.md`):

> If an asset loses its `.meta` file, any reference to that asset is broken in
> your project.

So a lost `.meta` orphans every material that used a texture and turns every
GameObject using a script into an "unassigned script component." **Commit every
`.meta` (never `*.meta` in `.gitignore`).** Meanwhile `Library/` is "a local
cache … unique to your computer" (`default-directories.md`) that Unity
regenerates on open — **never commit it.** New users routinely commit the
gigabyte `Library/` cache and forget the tiny `.meta` files. Do the opposite.

**The corollary CLI gotcha:** there is no offline "create a full C# project"
command. `-createProject` exists but needs an installed Editor, and the
`.csproj`/`.sln`/`Library/` are *generated by the Editor on first open*. So the
move is: write the portable on-disk skeleton (Assets/, Packages/manifest.json,
ProjectSettings/, .meta, a sample .asmdef) — which `new_unity_project.sh` does
with no Editor — then open once in Unity 6 (or `unity.sh -batchmode … -quit`) to
generate the rest. Don't try to coax the Editor into producing a `.csproj`
headlessly from scratch; produce the skeleton, then let it open.

## Creating a new project

```bash
scripts/new_unity_project.sh <dir> <ProjectName>
```

Runs **without an Editor** — it only writes files. It produces:

| Path | Why it must be this |
|------|---------------------|
| `Assets/` + `Scripts/ Scenes/ Editor/ Art/ Audio/` | Source layout. `Editor` is a reserved folder — its scripts are stripped from Player builds (`SpecialFolders.md`). |
| `Assets/.../<Name>.asmdef` | Game-code assembly definition so editing one script doesn't recompile everything (`assembly-definitions-intro.md`). |
| `Assets/.../Hello.cs` (+ `.meta`) | Sample MonoBehaviour; class name == file name (required to attach as a component). |
| `Packages/manifest.json` | The UPM project manifest — core packages + engine modules. An empty manifest loads *no* packages (`upm-manifestPrj.md`). |
| `ProjectSettings/ProjectVersion.txt` | Pins the 6000.x Editor version (override with `UNITY_EDITOR_VERSION`). |
| every asset + folder `.meta` | So references survive version control. |
| `.gitignore` | Ignores `Library/ Temp/ UserSettings/ obj/` + generated `*.csproj`/`*.sln`; keeps `Assets/ Packages/ ProjectSettings/` and all `.meta`. |

Then finish setup by opening once in the Editor, or headless:

```bash
scripts/unity.sh -batchmode -nographics -quit -projectPath <dir> -logFile -
# generates Library/, the <Name>.csproj and the .sln, and fills out ProjectSettings/
```

## Running Unity from the command line

Call Unity through `scripts/unity.sh`, never the raw path — it resolves the
binary in one place (`$UNITY_PATH` → Hub `6000.*` dirs → `unity` on PATH). The
wrapper does **not** add `-batchmode`/`-nographics` for you; pass them yourself.
The manual's canonical CI invocation (`command-line-run-unity.md`):

```bash
scripts/unity.sh -batchmode -nographics -quit \
  -projectPath <dir> \
  -executeMethod MyEditorScript.PerformBuild \
  -logFile -
```

Key flags (all from `EditorCommandLineArguments.md`):

- `-batchmode` — no human interaction; exits **1** on any exception/failure.
- `-nographics` — no GPU; **turns off logs unless you add `-logFile`**.
- `-logFile -` — send the Editor log to **stdout** (essential under `-nographics`).
- `-quit` — quit after commands finish. **Don't combine with `-runTests`** (it
  quits before tests complete).
- `-projectPath <path>` / `-buildTarget <name>` — open this project / set platform.

The `-executeMethod` target **must be `static` and live in an `Editor` folder**
(it uses `UnityEditor`, which doesn't exist at runtime). Signal CI failure by
throwing or calling `EditorApplication.Exit(n)`. Full reference and the verbatim
`BuildPipeline.BuildPlayer` example: `references/cli_and_batchmode.md`.

## UPM packages — `Packages/manifest.json`

The Package Manager reads this on load. All keys are optional, but an empty
manifest loads no packages, so always keep a `dependencies` object:

```json
{
  "dependencies": { "com.unity.test-framework": "1.4.5", "com.unity.ugui": "2.0.0" },
  "enableLockFile": true,
  "resolutionStrategy": "lowest"
}
```

Dependency values can be a registry version, a `file:` path, a tarball, or a Git
URL. Embedded packages (a folder under `Packages/`) need no entry. Scoped
registries, resolution strategies, and the full key table:
`references/packages_and_asmdef.md`.

## Assembly definitions — `.asmdef`

By default all game scripts compile into one `Assembly-CSharp.dll`, so changing
any script recompiles all of them. An `.asmdef` carves a folder into its own
assembly to shrink recompiles and enforce module boundaries
(`assembly-definitions-intro.md`). The only required field is `name`; references
to other asmdefs go by name **or** `GUID:<guid>` (one form per array). Two traps:
**you cannot reference `Assembly-CSharp` from a custom asmdef**, and dropping an
asmdef on a folder that has an `Editor` subfolder pulls editor code into the
runtime assembly and breaks the build. Format and rules:
`references/packages_and_asmdef.md`.

## What to commit (and what never to)

Commit `Assets/`, `Packages/`, `ProjectSettings/`, and **every `.meta`**. Never
commit `Library/`, `Temp/`, `Logs/`, `UserSettings/`, `obj/`, or the generated
`*.csproj`/`*.sln`. The full table, the reserved-folder list, and the
visible-vs-hidden meta setting are in `references/project_layout_and_meta.md`;
the copy-paste ignore file is `assets/unity.gitignore`.

## When setup breaks

`references/troubleshooting.md` covers the failures that account for almost every
setup problem: `unity.sh` exit 127 (no Editor here), lost `.meta` / broken
references, an accidentally-committed `Library/`, batchmode "succeeded but built
nothing", asmdef "can't find the type", Package Manager loading nothing, and the
class-name-must-match-file-name rule.
